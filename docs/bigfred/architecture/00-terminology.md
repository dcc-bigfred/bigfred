## Terminology

This vocabulary is **authoritative** for the rest of the document, the
codebase (`pkgs/server/domain`, services, security policies), the REST
endpoints, the WebSocket protocol and the MCP tool names. Whenever the
domain meaning differs from the casual English word, the table below
wins.

| Term         | Polish / domain meaning                                                                                                                                                                       | Where it appears                                                                              |
|--------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------|
| **driver**   | *Maszynista* – a user who operates vehicles and trains. Owns the throttle for their own vehicles and for vehicles currently leased to them.                                                   | Role `driver`; UI "Driver mode"; security inputs `Actor` when applicable.                      |
| **signalman** | *Nastawniczy* – a user who directs train traffic from a signal box (interlocking), issues entry/departure decisions, and may **take over a driver's vehicle or train to re-park it** when operationally necessary. | Role `signalman`; `Interlocking`/`InterlockingSession`; `TakeoverRequest`.                    |
| **vehicle**  | A single rail vehicle that the system can control. Covers a **locomotive**, an **electric/diesel multiple unit (EMU/DMU)** *or* a **powered wagon** – anything with its own DCC address.       | `domain.Vehicle`; REST `/api/v1/vehicles`; WS `loco.*` (kept for DCC-protocol compatibility).  |
| **train**    | *Skład* – an ordered group of **1 or more vehicles** owned by a user, addressed and driven as a single unit. Trains can be leased like a single vehicle.                                       | `domain.Train`, `domain.TrainMember`, `domain.TrainLease`; REST `/api/v1/trains`; WS `train.*`. |
| **interlocking** | *Nastawnia* – a logical signal box. At most one signalman occupies a given interlocking at a time. Visibility is **scoped to the party** the driver is currently in (see *party*).            | `domain.Interlocking`, `domain.InterlockingSession`, `domain.PartyInterlocking`.               |
| **takeover** | A signalman taking driving authority over a driver's vehicle or train, with a 15-second window for the driver to reject.                                                                       | `domain.TakeoverRequest`; WS `takeover.*`.                                                     |
| **lease**    | A time-bounded permission to **drive** somebody else's vehicle or train. Lease **never grants edit rights**.                                                                                   | `domain.VehicleLease`, `domain.TrainLease`.                                                    |
| **radio**    | "Walkie-talkie" channel between drivers and signalmen, built on a closed vocabulary of standard phrases.                                                                                       | `domain.RadioMessage`, `domain.RadioPhrase`; WS `radio.*`.                                     |
| **party**    | *Impreza* – a modeling event. A logical room that users join after logging in; every party owns its own list of signalmen and its own whitelist of interlockings. **No end date.** Every **non-default** party is bound to **exactly one** layout. The system-provided **`default`** party is always present and is the **only** party allowed to have `LayoutID = NULL` – in `default` the driver picks the layout per drive session via a dropdown in the vehicle control view. | `domain.Party`, `domain.PartySignalman`, `domain.PartyInterlocking`; REST `/api/v1/parties`; WS `session.opened { partyId, layoutId? }`, `session.setLayout`. |
| **layout**   | *Makieta* – a physical model railway layout, paired with a connection definition (`loconet_serial` / `z21` / `loconet_tcp`). Non-default parties are bound to exactly one layout. In the `default` party the layout is chosen per session by the driver. Layouts are admin-only to edit.                       | `domain.Layout`, `domain.LayoutConnection`; REST `/api/v1/layouts`.                            |
| **function** | A single DCC function slot on a vehicle, in the range `F0`–`F32`. Carries a user-given **name**, an **icon** (closed catalogue), a **kind** (`latched` / `momentary`) and the DCC function number. A vehicle exposes any subset of the 33 slots; the lessee can invoke them but not edit. | `domain.VehicleFunction`, `domain.FunctionIcon`, `domain.FunctionKind`; REST `/api/v1/vehicles/{addr}/functions`. |
| **vehicle template** | *Szablon pojazdu* – a reusable function-list definition for a class of vehicles. Linked vehicles inherit the list live; the **first** edit a user makes on their vehicle's functions detaches it via copy-on-write (§3a.6). | `domain.VehicleTemplate`, `domain.TemplateFunction`; REST `/api/v1/vehicle-templates`.         |
| **script**   | *Skrypt* – a short **JavaScript (ES5.1+)** program authored by a user and **executed server-side** inside a sandboxed [Goja](https://github.com/dop251/goja) VM. The VM runs **inside the sibling `scripts-executor` process**, never inside `server`, so a misbehaving script cannot affect the main throttle. Attached to a single vehicle or a single train and surfaced on the throttle UI as an additional button with an icon from the function-icon catalogue. Every DSL call (`findFirstLoco`, `findByDCCAddr`, `setSpeed`, `funcOn/Off`, `sleep`, …) routes through the **same `LocoService` / `TrainService` and the same security policies as a manual throttle press**, so authorization, audit and the dead-man's switch contract are unchanged. | `domain.Script`, `domain.ScriptAttachment`; REST `/api/v1/scripts`; WS `script.run` / `script.stop` / `script.log` / `script.runStarted` / `script.runStopped`; RPC over `pkgs/server/executor`. |

Note on naming consistency: the **wire protocol** uses identifiers
prefixed with `loco.*` (e.g. `loco.setSpeed`) because at the DCC
protocol level "loco" is the established term and the existing
`pkgs/loco/commandstation.Station` interface speaks in `LocoAddr`. The
**application domain** prefers `vehicle` because it covers more than
locomotives (EMUs, powered wagons). Both terms refer to the same thing
when applied to a single rail vehicle.
